# dev-flow — 기획 → 구현 → 검증 워크플로우

> spec-kit으로 스펙을 먼저 고정하고, TDD로 구현하고, ECC 기준으로 코드 품질을 검증한다.

## 핵심 원칙: 단계 간 반드시 사용자 승인을 받는다

**각 STEP이 끝나면 반드시 멈추고 사용자에게 확인한다.**
사용자가 명시적으로 "진행해", "다음", "OK" 등 승인을 줄 때까지 다음 STEP으로 절대 이동하지 않는다.
사용자가 수정을 요청하면 현재 STEP을 다시 수행한다.

---

## STEP 1 · 스펙 작성 (spec-kit)

`specify` 명령어를 사용하기 전에, **먼저 사용자에게 역질문을 던져 요구사항을 명확히 한다.**

### 역질문 단계 (필수)

요구사항을 받으면 바로 구현하지 않고, 불명확한 항목을 골라 한 번에 질문한다:

- 어떤 사용자가 쓰는 기능인가? (인증 필요 여부)
- 입력값의 형식과 제약조건은?
- 예외 상황(엣지케이스)은 어떻게 처리하나?
- 기존 기능과 연동이 필요한가?
- 성능/보안 요구사항이 있나? (rate limiting, 암호화 등)
- 완료 기준이 무엇인가?

역질문 예시:
```
📋 스펙 작성 전 확인사항

Q1. 소셜 로그인(카카오/구글)도 포함인가요?
Q2. 비밀번호 조건이 있나요? (길이, 특수문자 등)
Q3. 이메일 인증이 필요한가요?
Q4. 로그인 실패 횟수 제한이 있나요?

답변해 주시면 스펙을 작성하겠습니다.
```

역질문 답변을 받은 후 `.spec.md` 파일을 생성한다.

1. 역질문 답변을 반영해 `.spec.md` 파일 작성
2. 스펙 포함 항목:
   - **목적**: 이 기능이 해결하는 문제
   - **입력/출력**: 명확한 인터페이스 정의
   - **엣지 케이스**: 예외 상황 목록
   - **완료 기준**: 언제 "완성"으로 볼 것인가

스펙 파일 위치: `{프로젝트루트}/specs/{기능명}.spec.md`

### ✋ STEP 1 완료 후 — 반드시 멈추고 확인

스펙 작성이 끝나면 아래 형식으로 사용자에게 보고하고 승인을 기다린다:

```
📋 STEP 1 완료 — 스펙 작성

[작성된 스펙 요약]
- 목적: ...
- 입력/출력: ...
- 엣지 케이스: ...
- 완료 기준: ...

수정할 내용이 있으면 말씀해 주세요.
없으면 "진행해" 라고 하시면 STEP 2 (TDD)로 넘어갑니다.
```

---

## STEP 2 · 테스트 먼저 작성 (TDD)

사용자 승인 후에만 시작한다.

구현 전에 테스트를 먼저 작성한다.

1. 스펙의 각 완료 기준 → 테스트 케이스로 변환
2. 테스트 실행 → **모두 실패** 확인 (Red)
3. 최소한의 코드로 테스트 통과 (Green)
4. 리팩터링 (Refactor)

### ✋ STEP 2 완료 후 — 반드시 멈추고 확인

```
🧪 STEP 2 완료 — 테스트 작성

작성된 테스트 케이스:
- [테스트 목록]

테스트 실행 결과: 전체 X개 실패 확인 (정상)

수정할 내용이 있으면 말씀해 주세요.
없으면 "진행해" 라고 하시면 STEP 3 (ECC 구현)으로 넘어갑니다.
```

---

## STEP 3 · 구현 (ECC 기준)

사용자 승인 후에만 시작한다.

ECC(`/Users/joojak/Desktop/AI_AGENT/ECC/`)의 코딩 표준을 적용한다.

```bash
node /Users/joojak/Desktop/AI_AGENT/ECC/scripts/ecc.js [언어]
```

ECC 적용 기준:
- 네이밍 컨벤션 준수
- 함수 크기 제한 (단일 책임)
- 주석 최소화 (코드가 스스로 설명)
- 에러 처리 명시

### ✋ STEP 3 완료 후 — 반드시 멈추고 확인

```
⚙️ STEP 3 완료 — 구현

구현된 내용:
- [변경/생성된 파일 목록]
- [주요 구현 내용 요약]

테스트 재실행 결과: 전체 X개 통과

수정할 내용이 있으면 말씀해 주세요.
없으면 "진행해" 라고 하시면 STEP 4 (full-review)로 넘어갑니다.
```

---

## STEP 4 · 검증 (full-review)

사용자 승인 후에만 시작한다.

구현 완료 후:

1. **테스트 전체 통과** 확인
2. **full-review 스킬** 호출해 코드 품질 점검
3. 스펙과 구현 일치 여부 최종 확인
4. CRITICAL/HIGH 발견 시 → 수정 후 재검증

### ✋ STEP 4 완료 후 — 최종 보고

```
✅ STEP 4 완료 — full-review 결과

[심각도별 이슈 목록]
CRITICAL: X건
HIGH: X건
MEDIUM: X건
LOW: X건

[CRITICAL/HIGH가 있으면]
→ 수정이 필요합니다. 수정 후 재검증하겠습니다.

[이슈 없으면]
→ 모든 단계 완료. 머지 가능합니다.
```

---

## 흐름 요약

```
요구사항
  ↓
[STEP 1] spec-kit → .spec.md 작성
  ↓ ✋ 사용자 승인 대기
[STEP 2] TDD → 테스트 먼저 → 실패 확인
  ↓ ✋ 사용자 승인 대기
[STEP 3] ECC → 표준 기반 구현
  ↓ ✋ 사용자 승인 대기
[STEP 4] full-review → 코드 품질 점검
  ↓
완료
```
