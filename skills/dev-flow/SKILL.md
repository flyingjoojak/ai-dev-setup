# dev-flow — 기획 → 구현 → 검증 워크플로우

> spec-kit(specify → plan → tasks)으로 기획을 고정하고, TDD로 구현하고, ECC 기준으로 코드 품질을 검증한다.

## 핵심 원칙: 단계 간 반드시 사용자 승인을 받는다

**CRITICAL: 각 STEP이 끝나면 반드시 아래 GATE 형식을 출력하고 멈춘다.**

```
⏸ GATE [STEP N] — 사용자 승인 필요
"진행해" / "다음" / "OK" 중 하나를 입력해야 다음 단계로 넘어갑니다.
```

- 사용자의 명시적 승인 없이 다음 STEP으로 이동하는 것은 **FORBIDDEN**
- 승인 키워드: "진행해", "다음", "OK", "응", "ㅇㅇ", "계속"
- 사용자가 수정을 요청하면 현재 STEP을 다시 수행한다
- GATE를 건너뛰었다면 즉시 현재 STEP으로 되돌아가 GATE를 출력한다

---

## STEP 1 · 스펙 작성 (specify)

`specify` 명령어를 사용하기 전에, **먼저 사용자에게 역질문을 던져 요구사항을 명확히 한다.**

### 역질문 단계 (필수)

요구사항을 받으면 바로 구현하지 않고, 불명확한 항목을 골라 한 번에 질문한다:

- 어떤 사용자가 쓰는 기능인가? (인증 필요 여부)
- 입력값의 형식과 제약조건은?
- 예외 상황(엣지케이스)은 어떻게 처리하나?
- 기존 기능과 연동이 필요한가?
- 성능/보안 요구사항이 있나? (rate limiting, 암호화 등)
- 완료 기준이 무엇인가?

역질문 예시:
```
📋 스펙 작성 전 확인사항

Q1. 소셜 로그인(카카오/구글)도 포함인가요?
Q2. 비밀번호 조건이 있나요? (길이, 특수문자 등)
Q3. 이메일 인증이 필요한가요?
Q4. 로그인 실패 횟수 제한이 있나요?

답변해 주시면 스펙을 작성하겠습니다.
```

역질문 답변을 받은 후 `.spec.md` 파일을 생성한다.

1. 역질문 답변을 반영해 `.spec.md` 파일 작성
2. 스펙 포함 항목:
   - **목적**: 이 기능이 해결하는 문제
   - **입력/출력**: 명확한 인터페이스 정의
   - **엣지 케이스**: 예외 상황 목록
   - **완료 기준**: 언제 "완성"으로 볼 것인가

스펙 파일 위치: `{프로젝트루트}/specs/{기능명}.spec.md`

### ✋ STEP 1 완료 후 — GATE (필수 출력)

```
📋 STEP 1 완료 — 스펙 작성

[작성된 스펙 요약]
- 목적: ...
- 입력/출력: ...
- 엣지 케이스: ...
- 완료 기준: ...

수정할 내용이 있으면 말씀해 주세요.

⏸ GATE [STEP 1] — 사용자 승인 필요
"진행해" / "다음" / "OK" 중 하나를 입력해야 STEP 2 (구현 계획)로 넘어갑니다.
```

---

## STEP 2 · 구현 계획 (plan)

사용자 승인 후에만 시작한다.

`plan` 명령어로 스펙을 기반으로 **어떻게 만들 것인가**를 결정한다.

1. `.spec.md`를 읽고 기술 구현 방향 결정
2. `plan.md` 작성 포함 항목:
   - **아키텍처**: 전체 구조와 레이어 설계
   - **기술 스택**: 사용할 라이브러리/프레임워크
   - **API 설계**: 엔드포인트, 인터페이스
   - **데이터 모델**: 스키마, 관계
   - **제약조건**: 성능/보안/운영 고려사항

계획 파일 위치: `{프로젝트루트}/specs/{기능명}.plan.md`

### ✋ STEP 2 완료 후 — GATE (필수 출력)

```
🗺️ STEP 2 완료 — 구현 계획

[계획 요약]
- 아키텍처: ...
- 주요 기술: ...
- API 설계: ...

수정할 내용이 있으면 말씀해 주세요.

⏸ GATE [STEP 2] — 사용자 승인 필요
"진행해" / "다음" / "OK" 중 하나를 입력해야 STEP 3 (태스크 분해)로 넘어갑니다.
```

---

## STEP 3 · 태스크 분해 (tasks)

사용자 승인 후에만 시작한다.

`tasks` 명령어로 스펙 + 계획을 **실행 가능한 단위**로 쪼갠다.

1. `.spec.md` + `plan.md`를 읽고 태스크 목록 생성
2. `tasks.md` 작성 규칙:
   - 각 태스크는 독립적으로 구현·테스트 가능한 단위
   - 순서 명시 (의존 관계 고려)
   - 각 태스크에 완료 기준 포함

태스크 파일 위치: `{프로젝트루트}/specs/{기능명}.tasks.md`

### ✋ STEP 3 완료 후 — GATE (필수 출력)

```
📝 STEP 3 완료 — 태스크 분해

태스크 목록 (총 N개):
1. [태스크 1] — 완료 기준: ...
2. [태스크 2] — 완료 기준: ...
...

수정할 내용이 있으면 말씀해 주세요.

⏸ GATE [STEP 3] — 사용자 승인 필요
"진행해" / "다음" / "OK" 중 하나를 입력해야 STEP 4 (TDD)로 넘어갑니다.
```

---

## STEP 4 · 테스트 먼저 작성 (TDD)

사용자 승인 후에만 시작한다.

`tasks.md`의 각 태스크를 TDD 방식으로 구현한다.

1. 태스크의 완료 기준 → 테스트 케이스로 변환
2. 테스트 실행 → **모두 실패** 확인 (Red)
3. 최소한의 코드로 테스트 통과 (Green)
4. 리팩터링 (Refactor)

### ✋ STEP 4 완료 후 — GATE (필수 출력)

```
🧪 STEP 4 완료 — 테스트 작성

작성된 테스트 케이스:
- [테스트 목록]

테스트 실행 결과: 전체 X개 실패 확인 (정상)

수정할 내용이 있으면 말씀해 주세요.

⏸ GATE [STEP 4] — 사용자 승인 필요
"진행해" / "다음" / "OK" 중 하나를 입력해야 STEP 5 (ECC 구현)으로 넘어갑니다.
```

---

## STEP 5 · 구현 (ECC 기준)

사용자 승인 후에만 시작한다.

ECC(`~/.claude/rules/ecc/`)의 코딩 표준을 적용한다.
(setup.sh로 설치 시 자동으로 Claude Code에 주입됨)

ECC 적용 기준:
- 네이밍 컨벤션 준수
- 함수 크기 제한 (단일 책임)
- 주석 최소화 (코드가 스스로 설명)
- 에러 처리 명시

### ✋ STEP 5 완료 후 — GATE (필수 출력)

```
⚙️ STEP 5 완료 — 구현

구현된 내용:
- [변경/생성된 파일 목록]
- [주요 구현 내용 요약]

테스트 재실행 결과: 전체 X개 통과

수정할 내용이 있으면 말씀해 주세요.

⏸ GATE [STEP 5] — 사용자 승인 필요
"진행해" / "다음" / "OK" 중 하나를 입력해야 STEP 6 (full-review)로 넘어갑니다.
```

---

## STEP 6 · 검증 (full-review)

사용자 승인 후에만 시작한다.

**CRITICAL: 아래 6개 에이전트를 반드시 모두 병렬 실행해야 한다. 6개 미만 실행은 STEP 6 미완료로 간주한다.**

| 에이전트 | 관점 |
|---------|------|
| `code-reviewer` | 정확성·재사용성·로직 오류 |
| `security-reviewer` | OWASP Top 10·인증·시크릿 노출 |
| `database-reviewer` | N+1·인덱스·트랜잭션·SQL 인젝션 |
| `silent-failure-hunter` | 에러 삼킴·빈 catch·로깅 누락 |
| `performance-optimizer` | 불필요한 루프·메모리 누수·렌더링 |
| `a11y-architect` | aria·키보드 탐색·색상 대비 |

구현 완료 후:

1. **테스트 전체 통과** 확인
2. **위 6개 에이전트를 한 번의 응답에서 병렬 실행** (독립적이므로 동시에 띄운다)
3. 스펙과 구현 일치 여부 최종 확인
4. CRITICAL/HIGH 발견 시 → 수정 후 재검증

### ✋ STEP 6 완료 후 — 최종 보고 (필수 출력)

```
✅ STEP 6 완료 — full-review 결과

에이전트 실행 체크리스트:
- [x] code-reviewer 완료
- [x] security-reviewer 완료
- [x] database-reviewer 완료
- [x] silent-failure-hunter 완료
- [x] performance-optimizer 완료
- [x] a11y-architect 완료
6/6 완료

[심각도별 이슈 목록]
CRITICAL: X건
HIGH: X건
MEDIUM: X건
LOW: X건

[CRITICAL/HIGH가 있으면]
→ 수정이 필요합니다. 수정 후 재검증하겠습니다.

[이슈 없으면]
→ 모든 단계 완료. 머지 가능합니다.
```

> **체크리스트에 [ ] (미완료) 항목이 있으면 해당 에이전트를 즉시 실행하고 결과를 보고한다.**

## 흐름 요약

```
요구사항
  ↓
[STEP 1] specify → 역질문 + .spec.md 작성
  ↓ ✋ 사용자 승인 대기
[STEP 2] plan → 아키텍처·기술 스택 결정 (plan.md)
  ↓ ✋ 사용자 승인 대기
[STEP 3] tasks → 실행 단위로 분해 (tasks.md)
  ↓ ✋ 사용자 승인 대기
[STEP 4] TDD → 테스트 먼저 → 실패 확인
  ↓ ✋ 사용자 승인 대기
[STEP 5] ECC → 표준 기반 구현
  ↓ ✋ 사용자 승인 대기
[STEP 6] full-review → 코드 품질 점검
  ↓
완료
```
