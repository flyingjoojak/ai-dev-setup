# full-review — 6개 리뷰어 종합 점검

> 한마디면 6개 전문 리뷰어가 같은 코드를 각자 관점에서 점검하고, 결과를 하나의 우선순위 리포트로 합친다.

---

## STEP 1 · 대상 확정

리뷰 대상을 정한다 (우선순위):
1. 사용자가 파일/폴더를 지정했으면 → 그 대상
2. 지정 안 했으면 → **최근 변경분** (`git diff` / `git status`로 수정된 파일)
3. git이 없거나 변경분이 없으면 → 사용자에게 "어느 파일/폴더를 리뷰할까요?" 한 번 물음

---

## STEP 2 · 6개 리뷰어 병렬 실행

**한 번의 응답에서 6개 에이전트를 동시에 띄운다** (독립적이므로 병렬 실행).

각 에이전트는 아래 관점 하나만 담당:

| 에이전트 | 관점 | 주요 체크 포인트 |
|---------|------|----------------|
| **code** | 정확성·재사용성 | 버그, 로직 오류, 중복 코드, 불필요한 복잡성 |
| **security** | 보안 | OWASP Top 10, 인젝션, 인증/권한, 시크릿 노출 |
| **database** | DB | N+1, 인덱스 누락, 트랜잭션, SQL 인젝션 |
| **silent-failure** | 무음 실패 | 에러 삼킴, 빈 catch, 실패 무시, 로깅 누락 |
| **performance** | 성능 | 불필요한 루프, 메모리 누수, 불필요한 렌더링 |
| **a11y** | 접근성 | aria, 키보드 탐색, 색상 대비, 스크린리더 |

에이전트 프롬프트 형식:
```
너는 [관점] 전문 리뷰어다.
아래 코드를 [관점] 관점에서만 점검하고, 발견 사항을 심각도(CRITICAL/HIGH/MEDIUM/LOW)와 함께 리스트로 반환해라.
해당 없으면 "해당 없음"이라고만 써라.

[코드]
```

---

## STEP 3 · 결과 통합

6개 에이전트 결과를 받은 후:

1. **심각도별로 병합** (CRITICAL → HIGH → MEDIUM → LOW)
2. 중복 발견 사항은 하나로 합침
3. CRITICAL/HIGH가 있으면 **수정 우선순위 목록** 추가

출력 형식:
```
## 종합 리뷰 리포트

### CRITICAL (즉시 수정)
- [security] XSS 취약점: line 42 innerHTML에 미검증 사용자 입력 직접 삽입

### HIGH (이번 PR 전 수정)
- [database] N+1 쿼리: getUserPosts()에서 루프 내 DB 호출
- [silent-failure] line 87 catch 블록이 에러를 삼키고 있음

### MEDIUM (다음 이터레이션)
- [code] formatDate() 함수가 3곳에 중복 정의됨
- [performance] 불필요한 전체 배열 재렌더링

### LOW / 해당 없음
- [a11y] 버튼에 aria-label 없음
- [database] 해당 없음

### 수정 우선순위
1. line 42 XSS 취약점 → sanitize 또는 textContent 사용
2. getUserPosts() N+1 → include/join으로 교체
3. catch 블록 에러 로깅 추가
```
